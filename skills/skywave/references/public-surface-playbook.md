# Public Surface Playbook

Use this playbook for public-facing websites, marketing surfaces, brand sites, local-business sites, and content-led product surfaces.

## Default Rule

Assume the site needs:

- clear information architecture
- strong UI/UX
- conversion-aware sections
- blog/content support unless excluded
- schema/entity coverage
- SEO/AEO/GEO planning
- security and privacy planning
- performance and accessibility expectations
- analytics and conversion tracking design

Do not jump directly into code before these are at least sketched.

## Required Output Order

1. Brief
   - audience
   - intent
   - market
   - business goal
2. Information architecture
   - sitemap
   - page list
   - user paths
3. UX/UI direction
   - visual direction
   - trust pattern
   - CTA strategy
4. Page plan
   - section inventory
   - page purpose
   - conversion role
5. Content surface plan
   - core pages
   - blog or resource center
   - FAQ surfaces
   - author or team surfaces when relevant
6. Schema inventory
7. SEO/AEO/GEO plan
8. Analytics and conversion plan
9. Security and privacy plan

## CRO and Trust Blocks

Public surfaces should consider:

- clear hero promise
- trust proof
- service or product credibility
- FAQ or objections handling when real user intent supports it
- clean CTA hierarchy
- contact and legitimacy signals
- form trust and abuse resistance
- transparent privacy or consent expectations when tracking is used

Use only the trust blocks the business can actually support.

## Blog and Content System

Treat blog or resource content as default for public sites unless:

- the project is internal
- the surface is a closed dashboard
- the user explicitly excludes blog/content

Blog integration should define:

- who the content is for
- what search intent it supports
- how it links back to conversion pages
- which entities and topics it strengthens

## Accessibility and Performance

Treat these as default quality bars, not optional polish:

- mobile-first responsiveness
- clear semantic structure
- keyboard-friendly interaction
- sensible loading behavior
- image and media discipline
- section density that does not crush readability

Do not recommend visually heavy ideas that conflict with Core Web Vitals unless the tradeoff is explicit.

## Analytics and Conversion

Before release, define:

- primary conversion events
- secondary engagement events
- form success signals
- CTA instrumentation
- content-to-conversion linking points

Do not add tracking blindly. Keep it intentional and aligned with user goals.

## Exceptions

You may reduce the public-surface layers when the project is:

- an internal tool
- an admin dashboard
- a private workflow app
- a backend-only service

In those cases, keep the planning lean and do not force blog or heavy public-content behavior.
